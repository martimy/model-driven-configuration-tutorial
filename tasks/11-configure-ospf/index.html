Configure OSPF

This task covers configuring OSPF using NETCONF following the typical workflow of module discovery, payload editing, and loading.

OSPF Model Support

http://openconfig.net/yang/ospfv2?module=openconfig-ospfv2&revision=2024-06-17
urn:nokia.com:srlinux:ospf:ospf?module=srl_nokia-ospf&revision=2025-10-31

http://openconfig.net/yang/ospf?module=openconfig-ospf&revision=2025-02-13

Configuration

One of the challenges in model-driven configuration is the lack of documentation. Most vendor documents focus on the CLI, so finding a clear instructions on how to configure OSPF using NETCONF was a challenge.

However, SR Linux is built to support YANG data models. Therefore, all management interfaces (CLI, gNMI, and JSON-RPC) are based on a common YANG models for configuration, state, and operational tasks. For example, the CLI command tree is derived from the YANG models loaded into the system and a gNMI client can use the models to configure the system.

Therefore, using SR Linux documents has been helpful in discovering the requirements for configuring OSPF in SR Linux. From the documentation, we find that we need the following to create a basic OSPF configuration:

  • OSPF version number (v2 in this task)
  • Router ID: Each router running OSPF must be configured with a unique router ID.
  • An area: At least one OSPF area must be created. An interface must be assigned to each OSPF area.
  • Interfaces: An interface has a single IP address and mask (unless the network is an unnumbered point-to-point network).

Configuring all of the above will be done at the network instance level, so let's visit the network instance tree again:

pyang -f tree -p nokia  srl_nokia-network-instance.yang --tree-path /network-instance --tree-depth=2 --ignore-errors

You can also inspect the OSPF module tree, which is also large:

pyang -f tree -p nokia srl_nokia-ospf.yang

module: srl_nokia-ospf

  augment /srl_nokia-netinst:network-instance/srl_nokia-netinst:protocols:
    +--rw ospf!
       +--rw instance* [name]
          +--rw name                           srl_nokia-comm:name
          +--rw admin-state?                   srl_nokia-comm:admin-state
          +--rw version                        identityref
          +--rw address-family?                identityref
          +--rw instance-id?                   uint32
          +--rw router-id?                     yang:dotted-quad
          +--rw max-ecmp-paths?                srl_nokia-ospf-types:ospf-ecmp
          +--rw advertise-router-capability?   enumeration
          +--rw export-policy?                 -> /srl-rpol:routing-policy/policy/name
          +--rw asbr!
          ...

It is worth mentioning that the ! symbol immediately following a node name (such as ospf!) indicates that the node is a presence container. Unlike a normal container, a presence container exists in the configuration just by being created, even if it has no child nodes. Its mere existence signifies that OSPF is enabled in the network instance.

Typically, what you need to do next, is to generate the XML skeleton file that include the minimum configuration items mentioned above. However, for such configuration, relying on another technique that we explored earlier in Task 10, may be more efficient.

./nc_wrapper.sh srl-01 --get-config --filter /network-instance[name=default]

<?xml version='1.0' encoding='UTF-8'?>
<data xmlns="urn:ietf:params:xml:ns:netconf:base:1.0" xmlns:nc="urn:ietf:params:xml:ns:netconf:base:1.0">
    <network-instance xmlns="urn:nokia.com:srlinux:net-inst:network-instance">
        <name>default</name>
        <description>Default</description>
        <interface>
            <name>2</name>
            <interface-ref>
                <interface>ethernet-1/1</interface>

            </interface-ref>
        </interface>

cEOS

pyang -f tree -p openconfig --tree-path network-instances/network-instance/protocols/protocol/ospfv2 openconfig-network-instance.yang > ospf.txt

From the generated tree, we are interested in these tree branches and mandatory elements (router id will be assigned automatically if not provided):

module: openconfig-network-instance
  +--rw network-instances
     +--rw network-instance* [name]
        +--rw protocols
           +--rw protocol* [identifier name]
              +--rw ospfv2
                 +--rw global
                 |  +--rw config
                 |  |  +--rw router-id?                    yang:dotted-quad
                 ...
                 +--rw areas
                    +--rw area* [identifier]
                       +--rw identifier       -> ../config/identifier
                       +--rw config
                       |  +--rw identifier?   oc-ospf-types:ospf-area-identifier
                       ...
                       +--rw interfaces
                       |  +--rw interface* [id]
                       |     +--rw id               -> ../config/id
                       |     +--rw config
                       |     |  +--rw id?                             oc-if:interface-id
                       |     |  +--rw network-type?                   identityref
                       ...

Generate a XML skeleton at a subtree: 

pyang -f sample-xml-skeleton --sample-xml-skeleton-path=/network-instances/network-instance/protocols/protocol/ospfv2 --sample-xml-skeleton-doctype=config -p openconfig openconfig-network-instance.yang --ignore-errors -o ospf.xml

Edit the file ospf and remove the elements that are not required:

<network-instances xmlns="http://openconfig.net/yang/network-instance">
<network-instance>
    <protocols>
    <protocol>
        <ospfv2>
        <global>
            <config>
            <router-id>2.2.2.2</router-id>
            </config>
        </global>
        <areas>
            <area>
            <identifier>0.0.0.0</identifier>
            <config>
                <identifier>0.0.0.0</identifier>
            </config>
            <interfaces>
                <interface>
                <id>ethernet-1/1.0</id>
                <config>
                    <id>ethernet-1/1.0</id>
                </config>
                </interface>
                <interface>
                <id>ethernet-1/2.0</id>
                <config>
                    <id>ethernet-1/2.0</id>
                </config>
                </interface>
            </interfaces>
            </area>
        </areas>
        </ospfv2>
    </protocol>
    </protocols>
</network-instance>
</network-instances>
